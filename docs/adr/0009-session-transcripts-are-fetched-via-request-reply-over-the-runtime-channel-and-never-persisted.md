# Session Transcripts are fetched via request/reply over the runtime channel, and never persisted

ADR-0008 makes the Session Transcript — the agent-persisted conversation history on disk — the source of truth for what was said. The detail page needs to render it, but it lives inside a running Devcontainer and the existing runtime channel cannot carry it. The channel has exactly two patterns: a fire-and-forget `command` (Control Plane → Runtime, no reply) and a `runtime_event` (Runtime → Control Plane, persisted and projected, ADR-0002). A transcript fits neither. It cannot be a Runtime Event — those are low-volume structured facts that get persisted and projected into read models, and a transcript is neither low-volume nor a fact to project (it would dump large blobs into `runtime_events` and corrupt the projection's meaning). It cannot be a plain Command — the Control Plane needs the bytes *back*, correlated to the specific HTTP request that asked.

We add a **third channel pattern: request/reply with a correlation id**, multiplexed over the same per-Devcontainer agent WebSocket (no new connection). The Control Plane sends `transcript_request{request_id, agent_session_id}`; the agent answers `transcript_response{request_id, turns}`. The HTTP handler registers an `asyncio.Future` keyed by `request_id`, sends the request, and awaits it; the agent connection's read-loop resolves the matching Future when the response arrives. `AgentRegistry` grows one send-and-await method. The `request_id` is Control-Plane-generated and **never persisted** — it is ephemeral RPC, not a Runtime Event — which keeps ADR-0002's "all read-model state is a projection of `runtime_events`" invariant exactly intact.

The Control Plane **does not store the transcript**: it proxies the fetch live and keeps nothing. This mirrors the deliberate non-persistence of Session Output and avoids introducing a second blob store and the staleness it would carry as a conversation grows across resumes. The honest cost is that a full transcript is only available while the Devcontainer is **running** (agent connected). We accept it because the headline feature — continuing the conversation — already requires a running container, and the stopped-state view degrades to the already-persisted Session Summary projection, so no information available today is lost. Snapshotting transcripts for stopped-state viewing remains a purely additive future change.

The agent **parses its own JSONL into normalized turns** before sending, rather than shipping raw Claude session files. The raw format is Claude Code's internal, version-specific representation (mixed `mode`/`permission-mode`/`file-history-snapshot`/`user`/`assistant`/`last-prompt` lines); only the agent is pinned to a Claude version, so it is the one place that should know that format. Consumers see a stable Vibing shape instead — `{role, blocks:[{kind:"text", text} | {kind:"tool_use", name, summary}], at}` — carrying conversation text plus collapsed tool markers (tool *name* and a short summary, not tool output). The shape leaves room to add `tool_result` blocks later without a wire break. The day Claude changes its file format, the fix is in the agent alone, never the Control Plane or frontend.

The failure contract is part of the decision, since request/reply introduces failure modes the existing patterns lack. If the Devcontainer is stopped or no agent is connected, the Control Plane does not send at all and returns immediately, so the frontend takes the Session Summary fallback. If the agent is connected but does not answer within a flat timeout (`VIBING_TRANSCRIPT_TIMEOUT_SECONDS`, default 10s), the Future is rejected and the caller gets an error state — the timeout is what stops a slow or oversized transcript from hanging a request forever. A session whose file does not exist yet (created but never run) is **not** an error: the agent replies with `turns: []` and the UI shows "no conversation yet." If an agent connection drops, every in-flight Future for it is rejected so no handler hangs.

We chose request/reply over the two alternatives. Forcing it onto command+event (a `fetch_transcript` Command answered by a `transcript_fetched` Runtime Event) was rejected because it persists transcript blobs and pollutes the projection. Giving the agent its own HTTP endpoint for the Control Plane to call was rejected because it inverts the star topology (ADR-0003) — agents are inside containers and connect outbound to the hub, they are not addressable servers.

Consequences: the runtime channel now carries three message patterns, and the Control Plane holds per-request in-flight state (a Futures map with timeout and disconnect cleanup) it did not before — modest, but real, connection-scoped state. Transcript availability is coupled to Devcontainer liveness. Very large transcripts are sent as a single message; chunking or pagination is deferred until a real size problem appears, bounded for now only by the timeout.

Status: accepted
