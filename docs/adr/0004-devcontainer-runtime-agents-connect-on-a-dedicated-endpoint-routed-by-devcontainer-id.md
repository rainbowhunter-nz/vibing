# Devcontainer Runtime Agents connect on a dedicated endpoint, routed by devcontainer_id

ADR-0003 committed to a star topology with a single connected Host Runtime Worker per Control Plane, and explicitly deferred runtime identity and routing: "Accepting multiple host workers would require stable runtime identity and explicit Devcontainer-to-runtime ownership… that routing model is unnecessary while Vibing is local-only." The Devcontainer Runtime Agent forces that question now. Unlike the host worker, agents are inherently plural: one runs inside each *running* Devcontainer, and a `start_agent_session` Command must reach the agent for *that* Devcontainer — not the host worker, not another Devcontainer's agent. The single-slot connection model cannot express this.

We add a **second runtime WebSocket endpoint, `/runtime/agent/ws`**, distinct from the host worker's `/runtime/ws`, and route agent Commands by `devcontainer_id`. The Control Plane keeps the existing single host-worker slot (`RuntimeConnectionManager`) untouched and adds an `AgentConnectionManager` holding a `dict[devcontainer_id, WebSocket]`. An agent registers by sending the existing `runtime_registered` envelope with `source="devcontainer_runtime_agent"` and its `devcontainer_id` (a new optional field on `RegisterEnvelope`; the host worker leaves it null). A registration without a `devcontainer_id`, or a second agent for a `devcontainer_id` that already has one, is rejected — mirroring the host worker's single-slot rejection and matching the MVP rule of one active agent-session per Devcontainer. Inbound RuntimeEvents from both endpoints flow through the same `persist_runtime_event` intake; only connection tracking differs.

We chose a **separate endpoint over multiplexing both runtime kinds on one channel** (registry keyed by `(source, devcontainer_id)`). Multiplexing is more uniform but rewrites the proven single-slot host-worker logic and its tests for no behavioural gain; the two kinds have genuinely different connection semantics (one host worker vs. N per-Devcontainer agents), so separating them at the transport layer keeps each manager small and the host path a no-op change.

Agents are **launched by the Host Runtime Worker**, not supervised by the Control Plane and not declared in the user's `devcontainer.json`. After a successful `devcontainer up`, the worker fires a detached, best-effort `devcontainer exec` that starts `vibing devcontainer-runtime`, injecting `--devcontainer-id <id>` and `--control-plane-url ws://host.docker.internal:8000/api/v1/runtime/agent/ws` (the agent reaches the Control Plane from *inside* the container, hence `host.docker.internal`, not `127.0.0.1`). This is the only place the Control Plane's Devcontainer id — a backend-owned identifier nothing inside the container otherwise knows — crosses into the container. Launch is best-effort: a failure logs a warning and leaves the emitted `devcontainer_started` intact; a missing agent surfaces later as a `409` on `start_agent_session`, exactly as a missing host worker does today. This keeps the Control Plane out of in-container process management while the agent protocol is still being proven, consistent with ADR-0003's stance on the host worker.

The shared transport — reconnect/backoff loop, registration, command parsing, and the event-emit closure — lives in `src/vibing_runtime_client` and is consumed by both runtimes, rather than duplicated. The host worker is refactored onto it. The two runtimes still differ above the transport: the agent registers with a `devcontainer_id` and its command consumer is non-blocking (a session runs as a background task so `stop_agent_session` stays serviceable), whereas the host worker runs lifecycle Commands to completion serially.

Consequences: there are now two runtime endpoints and two connection managers to keep coherent, and agent identity depends on the host worker correctly injecting the `devcontainer_id`. If an agent crashes mid-container-life nothing restarts it until the Devcontainer is restarted. The topology remains a star with no runtime-to-runtime communication; this ADR only widens the hub to address N per-Devcontainer agents alongside the one host worker. Routing across multiple hosts or multiple host workers remains out of scope (ADR-0003).

Status: accepted

## Amendment (2026-06-05): the runtime is injected via `docker cp`, not baked into the image

The original decision said the worker "fires a detached, best-effort `devcontainer exec` that starts `vibing devcontainer-runtime`" but left unstated *how `vibing` comes to exist inside the container*. In practice it only worked because the image had the wheel baked in (the `devcontainer_examples/sandbox` Dockerfile `COPY`s a wheel and runs `uv tool install`). That implicit requirement contradicts this ADR's own principle that the `devcontainer_id` is the *only* thing that should have to cross into the container: it forced every user to modify their project's devcontainer image to install a Vibing-internal package before Vibing could drive it — the chief adoption blocker for the MVP.

We now **inject the runtime at launch over `docker cp`** instead of requiring it pre-installed. After a successful `devcontainer up`, the Host Runtime Worker copies the `uv` binary (already present in the Control Plane image) plus the prebuilt `vibing` wheel into the container, then the same best-effort `devcontainer exec` runs `uv tool install --python 3.13 --from <wheel> vibing` (uv fetches a standalone CPython 3.13 and resolves dependencies from PyPI) before launching `vibing devcontainer-runtime …`. The container id needed for `docker cp` comes from the `containerId` already parsed out of the `devcontainer up` JSON. The Control Plane image gains a `uv build --wheel` step so a copyable artifact exists. The `docker cp` reuses the host worker's configured engine (`docker`/`podman`), consistent with the stop path that already shells the engine directly.

This narrows the container contract to what a coding devcontainer plausibly already has: the user's `devcontainer.json` must provide `claude` (installed and authenticated) and network egress, and on Linux the `--add-host=host.docker.internal:host-gateway` runArg — but **nothing Vibing-specific**. We chose runtime injection with online dependency resolution over the two alternatives: baking (rejected — pushes per-project image edits onto the user) and a self-contained zipapp/pex run by the container's own Python (rejected — assumes a compatible interpreter and matching arch for native deps like `pydantic-core`, which heterogeneous devcontainers won't guarantee). The uv-managed standalone Python removes any assumption about a container Python at the cost of a one-time download on cold start.

Consequences: cold-start now depends on the container reaching PyPI and the astral CDN, and on `docker cp` access to the engine the worker already drives. Injection remains best-effort exactly as before — a copy or install failure logs a warning and surfaces later as a `409` on `start_agent_session`. Baking is no longer required; `devcontainer_examples/sandbox` drops its wheel `COPY`/`uv tool install` and becomes a plain devcontainer (claude + host-gateway). Offline dependency resolution (a vendored wheelhouse) was considered and deferred — the agent needs network for the Anthropic API anyway, so it buys no resilience the MVP needs.
