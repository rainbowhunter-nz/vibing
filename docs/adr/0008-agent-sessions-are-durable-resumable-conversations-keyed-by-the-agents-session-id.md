# Agent Sessions are durable, resumable conversations keyed by the agent's own session id

The original model (CONTEXT.md, ADR-0002) treated an Agent Session as a *single run*: one `start_agent_session`, one prompt, a terminal end state (`completed`/`failed`/`stopped`), and a one-line Session Summary as the only durable record of what happened. But `claude -p` already persists a full, structured, **resumable** conversation to disk under `~/.claude/projects/<encoded-cwd>/<session-id>.jsonl`, and `claude --resume <session-id>` continues that exact thread. The single-run model throws that away — the detail page could only ever show the first prompt and the final result. The real product goal is to **view the actual conversation and continue it**, which the single-run framing cannot express.

We redefine an **Agent Session as the durable conversation itself, not one run**, and make its identity the agent's own session id. A new session launches with `--session-id <agent_session.id>` (our id is already a uuid4, which `--session-id` requires), so `agent_session.id` == the Claude session id == the name of the on-disk transcript file. This collapses three identifiers into one and removes any need to parse Claude's output or scan the `.claude` folder to correlate a session with its file: the Control Plane already knows the id and therefore the exact path. Continuing a conversation is a new **`resume_agent_session`** Command carrying the follow-up prompt; the agent runs `claude -p "<prompt>" --resume <id>`, which by default reuses the same session id and appends to the same file (`--fork-session` would branch — we do not use it).

The lifecycle stops being terminal. `completed`/`failed`/`stopped` become **resumable resting states**: `resume_agent_session` takes any of them back to `starting → running`, reusing the existing `agent_session_started` → `session_completed`/`session_failed`/`session_stopped` events with **no new event types**. The one-active-per-Devcontainer invariant is unchanged and now also gates resume — only one `claude` can run per container, so resume is rejected if another session is active. The single `prompt` column degrades to "the first message / a label"; the Session Transcript (see ADR-0009), not the database, is the source of truth for what was said across turns.

We chose this **"session = the conversation, re-opened in place"** over the alternative of **"session = one run, chained"** (each continuation a new `agent_session` row resuming the prior Claude id). Chaining keeps the lifecycle terminal and is a smaller conceptual change, but it splits one conversation across N rows, breaks `agent_session.id == Claude session id`, and forces the UI to stitch a chain back together — pushing complexity into every consumer to preserve an invariant (terminal states) that buys nothing once resume exists. Folding the conversation onto one stable identity matches the user's mental model ("the conversation I continue") and keeps id, file, and row in lockstep.

This forces one projection change. The Session Summary is written on every terminal event by a blind `INSERT`, which assumed exactly one completion per session; a re-openable session can complete repeatedly, so the summary write becomes **upsert-by-session** to preserve "one Session Summary per Agent Session." Each completion still emits a fresh COMPLETION Inbox Event — every finished run is worth a notification — so re-completion is observable without duplicating the summary. Hand-started sessions (a human running `claude` inside the container, with no `agent_session` row) are explicitly out of scope: they would require folder enumeration and an "adopt orphan session" identity story, which this decision deliberately avoids by only ever addressing sessions Vibing launched.

Consequences: "completed" now means "idle, resumable," not "done," and any code or copy that treated end states as final must be revisited. Sessions accrete history indefinitely on disk with no built-in pruning. The deterministic `--session-id` ties us to Claude Code's session-file convention, but that coupling is owned entirely by the Devcontainer Runtime Agent, which already owns "how Claude runs." The `started_at`/`ended_at` columns, already only loosely populated, describe at most the latest run, not the whole conversation.
