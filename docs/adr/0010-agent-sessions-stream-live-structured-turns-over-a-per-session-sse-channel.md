# Agent Sessions stream live structured turns over a per-session SSE channel; the transcript stays the source of truth

ADR-0009 made the Session Transcript the source of truth for what was said, fetched on demand and **never streamed**, and ADR-0005/0006 made the live `/api/v1/events` SSE **invalidation-only** (scope + ids, no payload). With that pair, a running Agent Session shows no progress until it pauses or ends: the agent runs as a single batch `claude -p --output-format json` call drained with `communicate()`, and the transcript only refetches when `session_status` changes. To make the conversation view a standard live chat — text typing in, tool cards appearing as the agent works — we need a payload-bearing live channel that those two ADRs deliberately excluded.

We introduce the **Session Stream**: a live, ephemeral, incremental projection of an *active* session's turns, distinct from both the raw-character Session Output (which we are not building) and the durable Session Transcript. It refines — does not overturn — ADR-0009 and ADR-0005/0006: the transcript remains the source of truth, the global invalidation SSE stays payload-free, and the stream is a transient view that yields to the transcript when a run ends.

**Pipeline.** The runner switches to incremental `claude -p --output-format stream-json --verbose --include-partial-messages` (composing with `--resume`/`--session-id` per ADR-0008), reading stdout line-by-line and invoking a delta callback; the terminal `result` event still drives the success/failure mapping that produces the `session_completed`/`session_failed` Runtime Events. The **devcontainer runtime** parses stream-json into normalized **turn-deltas** and emits them over its existing agent WebSocket as a new envelope — keeping all Claude-format coupling in the runtime, the same principle that pins `transcript.py` as the only reader of Claude's on-disk format. The Control Plane relays deltas to browsers over a **new per-session SSE endpoint**, separate from the invalidation `/events` stream so ADR-0005/0006's payload-free invariant is preserved.

**Replay without persistence.** The Control Plane holds an **in-memory per-run replay buffer** (with event ids for SSE `Last-Event-ID` reconnect): a browser connecting mid-run or refreshing replays the current run, then streams live. The buffer is evicted on the terminal event and **never persisted to disk** — honoring the "ephemeral, not a goal to persist" stance Session Output already carries, while surviving refresh within a run. After a run ends, a fresh load relies on the durable transcript, not the buffer.

**Reconciliation by shared id.** The live view is merged with the canonical transcript rather than swapped, so we add a stable **turn `id`** (Claude's per-message uuid) to the normalized `TranscriptTurn`, captured in `transcript.py` and emitted on every stream-delta. The frontend keys turns by id: partials update in place, and a transcript fetch reconciles by id with no duplicate bubbles or flicker. Without this id, merge would degrade to brittle content/positional matching exactly where text is changing.

**Considered and rejected.** Polling the transcript while a session is active (frontend-only, no backend change) was rejected because it updates at turn granularity at best and never gives token-level typing. Extending the global `/events` SSE to carry output payloads was rejected because it breaks the invalidation-only invariant and fans every session's output to every client. Streaming raw Session Output characters into a terminal pane was rejected because a standard chat renders structured messages and tool cards, not a console — and that raw channel maps to neither the transcript block model nor the chat UI. A browser-side WebSocket was rejected as bidirectional overkill for one-way output.

Consequences: the Control Plane gains a second live channel and per-run in-flight buffer state it did not have before, and the runtime channel carries a fourth message shape (alongside command, runtime_event, and transcript request/reply). The normalized turn shape grows an `id` field. Token-level partials raise event volume; the replay buffer coalesces partials to stay small. Inline interventions in the chat are built against `approval_requested`/`agent_asked_question` events that the agent does not yet emit (it runs `bypassPermissions`; detection is the deferred `--permission-prompt-tool` work), so that UI is mock-only until those events become real and may need to follow the eventual event shape.

Status: accepted
